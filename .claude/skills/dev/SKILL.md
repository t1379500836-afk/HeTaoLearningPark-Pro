# 开发工作流

按以下步骤执行开发任务：

## 0. 熟悉项目

新对话开始、刚接手项目、或对现状不确定时，**主动读取** `docs/` 下的文档了解项目全貌，再开始工作。不要凭猜测动手。

## 1. 理解需求

梳理用户要做什么，明确改动范围。如果需求不清晰，先提问再动手。

## 2. 设计方案

说明要改哪些文件、怎么改、为什么这样改。等用户确认后再编码。

涉及架构变更时，同步说明文档更新计划。

## 3. 编码实现

按 CLAUDE.md 中的架构原则和编码风格规范编写代码。

## 4. 自审

写完代码后逐项检查：

- 代码是否有语法错误？
- 是否遵循了解耦和简洁原则？
- 是否引入了不必要的依赖？
- 前端样式是否与现有风格一致？
- 课程内容是否在正确阶段范围内（不超纲、答案正确）？

发现问题直接修复，不遗留。

## 5. 更新文档

如果本次改动涉及以下情况，同步更新 `docs/` 目录：

- 新增/删除了功能模块 → 更新 `docs/features.md`
- 修改了目录结构、数据流、状态管理 → 更新 `docs/architecture.md`
- 修改了路由、配置机制 → 更新 `docs/architecture.md`

每次改动追加一条到 `docs/changelog.md`。

### 文档更新约束

- **只改对应部分**：更新时修改相关段落，不要全篇重写或追加冗余描述
- **不加废话**：一行能说清的不写两行，不加"目前"、"暂时"、"接下来"等时效词
- **删旧换新**：被取代的内容直接删除，不要加"旧版…"或"原来…"做对比
- **changelog 最多保留最近 20 条**：超过时删除最早的条目
- **单篇文档不超过 150 行**：超过时精简内容，删除过时描述
- **不确定改不改文档时，不改**

## 6. 交付

简要说明改了什么、为什么这样改。不重复罗列代码内容。
